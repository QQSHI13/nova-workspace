## DropTransfer CLI Project

**Location:** `/home/qq/.openclaw/workspace/projects/droptransfer-cli/`
**GitHub:** https://github.com/QQSHI13/droptransfer-cli

Built a Node.js CLI tool for sending files from terminal to browser via WebRTC P2P. Uses the same PeerJS + WebRTC stack as the web DropTransfer (https://qqshi13.github.io/droptransfer/) for compatibility.

### Key Technical Challenges

1. **PeerJS Node.js compatibility** — PeerJS checks for browser globals (`RTCPeerConnection`, `navigator`, `window`). Required polyfilling these in Node.js:
   - `RTCPeerConnection` from `wrtc` package
   - Mock `navigator` object with `userAgent`
   - Mock `window` global

2. **wrtc compilation issues** — The `wrtc` npm package requires native compilation. On WSL/Ubuntu needs:
   - `build-essential`
   - `python3`
   - But still had issues with `node-pre-gyp` not found

3. **Final solution** — Created browser environment polyfills at the top of the script before importing PeerJS:
   ```javascript
   global.navigator = { userAgent: 'node.js', platform: 'linux' };
   global.window = global;
   global.RTCPeerConnection = require('wrtc').RTCPeerConnection;
   // etc.
   ```

### Usage
```bash
cd ~/.openclaw/workspace/projects/droptransfer-cli
npm install
node droptransfer-cli.js myfile.txt
# Shows code like "abc123def"
# Receiver enters code at https://qqshi13.github.io/droptransfer/
```

### Architecture
- CLI creates PeerJS peer (gets random ID = code)
- Waits for browser to connect using that code
- Sends metadata → waits for 'ready' signal
- Streams file in 256KB chunks via WebRTC data channel
- Progress bar in terminal

### Files Created
- `droptransfer-cli.js` — Main CLI tool with mocked browser env
- `package.json` — Dependencies: peerjs, wrtc
- `qq` — Bash wrapper script
- `README.md` — Documentation

### Status
Working locally but deployment complex due to wrtc native deps. Considered alternatives:
- Simple HTTPS server on Pi (too much setup)
- WebSocket relay (simpler but not P2P)
- Tailscale (best for cross-network but extra dependency)

**Decision:** Stick with WebRTC/PeerJS approach for true P2P, accept build complexity.

## WebRTC / Networking Discussion

Discussed file transfer architectures with QQ:
- **WebRTC P2P** — Direct connection, needs signaling server (PeerJS cloud)
- **IPv6 direct** — Would work without relay, but mobile broadband blocks inbound
- **NAT hole punching** — Requires coordinated timing (both sides connect simultaneously)
- **Mobile networks** — CGNAT blocks all inbound, need relay or Tailscale

Key insight: IPv6 doesn't solve firewall issues — home routers and mobile carriers still block inbound by default.

**Best portable solution:** WebRTC with STUN/TURN + signaling relay (what PeerJS provides).
